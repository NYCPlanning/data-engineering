DE Tools — How It Works

A DigitalOcean droplet running four Docker containers behind nginx. Two services are user-facing: a Streamlit QA app and a Dagster pipeline UI. Everything is wired up by apps/docker-compose.yml.

Request Flow

Services at a Glance

Key Code, Annotated

nginx apps/nginx/scripts/docker-entrypoint.sh + http.conf.template
lines 13–19
Auth is set up at container start, not build time. Credentials come in as env vars, get hashed into .htpasswd via htpasswd -c.
lines 25–38
If NGINX_DOMAIN is set → HTTPS config (redirects port 80, enables SSL). Otherwise plain HTTP. Decided once at startup; changing it requires docker compose restart nginx.
proxy lines
Upgrade + Connection: upgrade headers are required for Streamlit's WebSocket connection. Without them the QA app loads but freezes.
# docker-entrypoint.sh
# ── Step 1: generate .htpasswd from env vars ──
if [ -n "$NGINX_AUTH_USER" ] && [ -n "$NGINX_AUTH_PASSWORD" ]; then
    htpasswd -cb /etc/nginx/.htpasswd "$NGINX_AUTH_USER" "$NGINX_AUTH_PASSWORD"
else
    htpasswd -cb /etc/nginx/.htpasswd "detools" "changeme"  # ← fallback default
fi

# ── Step 2: pick HTTP vs HTTPS config ──
if [ -n "$NGINX_DOMAIN" ]; then          # production: HTTPS
    envsubst '${NGINX_DOMAIN}' \
        < https.conf.template > /etc/nginx/conf.d/default.conf
else                                      # local: plain HTTP
    envsubst '${NGINX_SERVER_NAME}' \
        < http.conf.template  > /etc/nginx/conf.d/default.conf
fi
exec nginx -g 'daemon off;'

# ── http.conf.template: proxy rules ──
location /qaqc {
    auth_basic "DE Tools";
    auth_basic_user_file /etc/nginx/.htpasswd;

    proxy_pass http://qa-streamlit:8501/qaqc;  # service name resolution
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;  # WebSocket support
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 86400;               # 24h — long Streamlit sessions
}
location /dag {
    proxy_pass http://dagster:3000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 86400;
}
dagster / ingest apps/dagster/ingest/assets.py
make_ingest_asset()
A factory function. At module load time it's called once per YAML file in ingest_templates/, producing one Dagster asset for each dataset (e.g. ingest_doitt_buildingfootprints).
DynamicPartitionsDefinition
Partitions represent dataset versions (e.g. 24v3). They're not auto-discovered — someone must add a partition in the Dagster UI before materializing.
ingest() call
All the real work happens inside dcpy.lifecycle.ingest.run.ingest(). The asset just bridges Dagster's config + partition system to that function.
# One asset per YAML template — generated at module load
def make_ingest_asset(template_id, depends_on=None):
    deps = [AssetKey(f"ingest_{dep}") for dep in (depends_on or [])]

    @asset(
        name=f"ingest_{template_id}",
        deps=deps,
        partitions_def=DynamicPartitionsDefinition(name="ingest_version"),
        group_name="ingest",
    )
    def _ingest_asset(context, config: IngestConfig, local_storage):
        from dcpy.lifecycle.ingest.run import ingest

        partition_key = context.partition_key  # ← the version string
        ingest(
            dataset_id=template_id,
            mode=config.mode,
            latest=config.latest,    # push to /latest/ folder in S3?
            push=config.push,        # False in ingest_dev_job
        )
        return MaterializeResult(metadata={"version": partition_key, ...})

    return _ingest_asset

# Module-level: load all templates + validate deps
ingest_templates = list_ingest_templates()   # scans ingest_templates/
# validate_dependencies() and detect_circular_dependencies() run here
ingest_assets = [
    make_ingest_asset(t.name, template_deps[t.name])
    for t in ingest_templates
]
qa-streamlit apps/qa/src/index.py + shared/constants.py
PAGES dict
Maps sidebar display names → module folder names. Adding a new product page means: add an entry here, create src/pages/<name>/<name>.py, expose a function with the same name.
importlib lazy load
Pages are not imported at startup — only the selected page is loaded. This keeps cold-start time short even though there are 14 pages.
query_params
The current page is persisted in the URL (?page=pluto), so links are shareable and browser back/forward works.
# constants.py — the page registry
PAGES = {
    "Home":                        "home",
    "PLUTO":                       "pluto",
    "Developments DB":             "devdb",
    "Facilities DB":               "facdb",
    "Data Ingestion":              "ingest",
    # ... 9 more entries
}

# index.py — routing
def run():
    query_params = st.query_params.to_dict()
    name = query_params.get("page", "Home")  # URL-driven

    name = st.sidebar.selectbox("Select a page", list(PAGES.keys()),
                                index=page_list.index(name))
    st.query_params["page"] = name              # keep URL in sync

    # Lazy-load the selected page module
    dataset_module = importlib.import_module(
        f"pages.{PAGES[name]}.{PAGES[name]}"   # e.g. pages.pluto.pluto
    )
    page_fn = getattr(dataset_module, PAGES[name])  # fn same name as module
    page_fn()
deployment apps/dagster/Dockerfile + apps/scripts/deploy.sh
runtime install
dcpy is not baked into the image. The CMD runs uv pip install -e /app/repos/data-engineering every time the container starts, pulling from the mounted repo checkout. No image rebuild needed for dcpy changes — just restart the container.
deploy.sh asymmetry
qa-streamlit is pulled from a registry. dagster is built locally on the droplet. That means Dagster image updates require a docker compose build, while QA app updates only need a pull.
--path-prefix /dag
Tells the Dagster webserver that it's being served from a sub-path. Without this, its internal asset/link URLs break when proxied through nginx at /dag.
# Dockerfile — dcpy installed at runtime, not build time
FROM nycplanning/build-geosupport:latest

RUN mkdir -p /app/repos/data-engineering    # mount point placeholder
ENV DAGSTER_HOME=/opt/dagster/dagster_home
RUN mkdir -p ${DAGSTER_HOME}

CMD sh -c "uv pip install --system -e /app/repos/data-engineering \
           && dagster-webserver -h 0.0.0.0 -p 3000 -w workspace.yaml \
           --path-prefix /dag"       # ← critical for nginx sub-path

# deploy.sh — asymmetric update paths
docker compose pull qa-streamlit         # pre-built image from registry
docker compose build dagster             # built on the droplet

docker compose up -d --force-recreate \
    qa-streamlit dagster dagster-daemon  # nginx not restarted by default

# daemon has the same image as webserver but different CMD:
command: ["sh", "-c",
  "uv pip install --system -e /app/repos/data-engineering
   && dagster-daemon run"]

Gotchas

nginx config is generated once at container start — not hot-reloaded
The entrypoint script runs envsubst and writes default.conf before nginx launches. If you change NGINX_DOMAIN, NGINX_AUTH_USER, or NGINX_AUTH_PASSWORD, you must docker compose restart nginx (not just reload) for the new values to take effect. Changing from HTTP to HTTPS also requires a restart because a completely different template is rendered.
Ingest partitions are manual — there is no auto-discovery of versions
The partition definition is DynamicPartitionsDefinition(name="ingest_version"). Before you can materialize any ingest asset for a new dataset version, you must first add that partition in the Dagster UI (Partitions tab → Add partition) or via the CLI. Nothing will fail visibly if you forget — the asset just won't appear as runnable for that version.
dcpy is installed from the mounted repo at every container start — not from the image
Both dagster and dagster-daemon run uv pip install --system -e /app/repos/data-engineering in their startup CMD. This means: (a) if the repo mount is broken, the container silently starts but dcpy is missing; (b) any dcpy change takes effect on the next container restart without rebuilding the image. The QA app takes a different approach — it runs from a pre-built image pulled from the registry.
Production volume paths are hard-coded to /root/data-engineering
docker-compose.yml mounts /root/data-engineering and /root/product-metadata directly. For local development, docker-compose.local.yml overrides these with ${DATA_ENGINEERING_PATH:-..}. If you ever clone the repo to a different path on the droplet, you must update the volume paths in docker-compose.yml or the containers will silently mount the wrong (or missing) directory.
Streamlit requires WebSocket — if it loads but freezes, check the nginx Upgrade headers
Streamlit uses a persistent WebSocket connection after the initial HTTP handshake. nginx must forward Upgrade: $http_upgrade and Connection: "upgrade". These are present in the templates, but if you ever replace or regenerate the nginx config manually and drop these headers, the app will render an empty shell and appear to hang with no error message.
certbot only runs in the production Docker Compose profile
The certbot service has profiles: [production]. A bare docker compose up won't start it. Production start/deploy scripts use op run --env-file=env.1pw -- ./scripts/start.sh which doesn't explicitly pass --profile production either — certbot only runs if you've explicitly started it once with docker compose --profile production up -d certbot. Check with docker compose ps to confirm it's actually running.
The QA app "Data Ingestion" page is a legacy manual upload path
The Streamlit page at /qaqc?page=ingest lets you upload a raw file to S3 and trigger a dcpy library call. This is the pre-Dagster ingest workflow. The Dagster-based ingest assets (which run against YAML templates) are the current path. Both still work, but they write to different S3 locations and have different configs — don't mix them for the same dataset.
Adding a new ingest template requires a Dagster container restart
The list of ingest assets is built at module load time in ingest/assets.py — it scans ingest_templates/ once when the Python module is imported. Adding a new .yml file to that directory doesn't hot-reload into a running Dagster instance. You need to restart the dagster and dagster-daemon containers for the new asset to appear in the UI.
